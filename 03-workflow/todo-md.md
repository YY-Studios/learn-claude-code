# TODO.md 주도 개발 (작업 연속성 유지)

여러 세션에 걸쳐 작업하다 보면 Claude가 어디까지 했는지 맥락을 잃기 쉽습니다.  
`TODO.md`로 작업 진척도를 명시적으로 관리하면 `/clear`를 해도 연속성을 유지할 수 있습니다.

---

## 내장 Todo 시스템 vs TODO.md 파일

먼저 두 가지를 구분하는 게 중요합니다.

| | **내장 Todo (TodoWrite)** | **TODO.md 파일** |
|---|---|---|
| 방식 | Claude가 세션 내 자동 생성 | 직접 만드는 파일 |
| 지속성 | 세션 종료 시 사라짐 | Git에 남아서 영구 보존 |
| 용도 | 현재 세션 진행 상황 표시 | **다음 세션으로 맥락 전달** |

> 💡 내장 Todo는 Claude가 복잡한 작업(3단계 이상)을 수행할 때 자동으로 만들어서 진행 상황을 표시합니다. 별도로 설정할 필요 없어요.  
> 세션 간 연속성이 목적이라면 **TODO.md 파일**이 필요합니다.

---

## 실전 워크플로우

### 1. TODO.md 초기 구조

```markdown
# 프로젝트명 TODO

## 🔥 현재 진행 중
- [ ] 인증 API 엔드포인트 작성

## ✅ 완료
- [x] Supabase 스키마 설계
- [x] 프로젝트 초기 세팅

## 📋 다음 예정
- [ ] 프론트엔드 로그인 UI 연결
- [ ] RLS 정책 설정

## 📝 메모 (의사결정 이유)
- 2025-03-25: JWT 대신 Supabase Auth 사용 결정 — 별도 토큰 관리 불필요
```

---

### 2. 세션 시작 시

```
"TODO.md를 읽고 오늘 진행할 작업을 파악해서 Plan을 세워줘."
```

### 3. 작업 중

```
"방금 완료한 인증 API를 TODO.md에 체크해줘."
```

### 4. 세션 종료 전

```
"오늘 완료한 작업과 다음에 이어서 할 작업, 
 그리고 중요한 의사결정 내용을 TODO.md에 업데이트하고 Git 커밋해줘."
```

---

## 복리 효과 (세션이 쌓일수록 강해짐)

TODO.md 파일은 살아있는 실행 기록이 됩니다. 다음 세션에서 Claude는 무엇을 해야 하는지만 보는 게 아니라, 이미 완료된 작업과 그 방법을 함께 봅니다.

```
세션 1: 프로젝트 구조 파악 + 스키마 설계
세션 2: 세션 1의 완료 내용을 보고 패턴 이해 → API 작성
세션 3: 세션 1+2의 전체 맥락으로 → 더 빠르고 적은 질문으로 작업
```

## CLAUDE.md에 TODO 읽기 규칙 추가하기

매번 말하기 귀찮다면 CLAUDE.md에 넣어두세요.

```markdown
## 작업 규칙
- 세션 시작 시 항상 TODO.md를 읽고 현재 상태를 파악할 것
- 기능 완료 시마다 TODO.md를 업데이트할 것
- 중요한 기술 결정은 TODO.md 메모 섹션에 날짜와 함께 기록할 것
```

---

## 주의사항

- TODO.md가 너무 길어지면 CLAUDE.md와 똑같은 문제가 생깁니다. 완료 항목은 주기적으로 정리하거나 별도 파일(`DONE.md`)로 분리하세요.
- 의사결정 이유(Why)를 함께 적어두면 나중에 같은 질문을 Claude에게 다시 안 해도 됩니다.
