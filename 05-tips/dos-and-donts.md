# 잘 되는 것 & 안 되는 것 (Dos and Don'ts)

---

## ✅ 적극 권장하는 것 (Dos)

**Plan Mode 먼저**
`Shift + Tab`으로 항상 계획부터 확인하고 코드를 수정하게 하세요. 계획이 잘못됐을 때 `ESC`로 즉시 중단하는 것도 습관으로 만드세요.

**TDD 기반 짧은 호흡**
작은 변경 → 린트/테스트 → 커밋의 짧은 주기를 유지하세요. 이렇게 하면 Claude가 잘못된 방향으로 가더라도 되돌리기 쉽습니다.

**이미지로 맥락 전달**
UI 톤앤매너, 아키텍처 구조, 디자인 레퍼런스는 말보다 스크린샷을 드래그앤드롭해서 전달하세요. 텍스트로 설명하는 것보다 훨씬 정확합니다.

**에러 로그 통째로 넘기기**
본인의 해석을 빼고 터미널 에러 로그 전체를 복붙하세요. 요약하거나 해석하면 오히려 정보가 손실됩니다.

**구체적인 지시**
모호한 지시는 모호한 결과를 낳습니다. "테스트 코드 작성해줘" 보다 "사용자가 로그아웃했을 때의 엣지케이스를 커버하는 테스트 코드를 작성해줘. 목킹은 하지 마." 처럼 구체적으로 지시하세요.

**think hard 활용**
복잡한 아키텍처 결정이나 엣지케이스가 많은 문제에는 프롬프트 앞에 `think hard`를 붙이세요. Claude가 더 깊이 추론합니다.

```
"think hard about the edge cases in our authentication flow"
```

**작업 전 탐색 먼저**
큰 작업 시작 전에 Claude에게 먼저 관련 파일을 읽게 하고 아무것도 수정하지 말라고 명시하세요. 탐색과 구현을 분리하면 엉뚱한 파일 수정을 예방합니다.

```
"먼저 인증 관련 파일들을 읽고 구조를 파악해줘. 아직 아무것도 수정하지 마."
```

## ❌ 절대 피해야 할 것 (Don'ts)

**하나의 세션에서 여러 기능 개발하기**
컨텍스트가 오염되어 환각(Hallucination)이 발생합니다. 한 세션 = 한 피처 원칙을 지키고, 기능이 끝나면 `/clear`하세요. 관련 없는 질문을 중간에 끼워넣는 것도 마찬가지입니다.

**대용량 파일 직접 분석시키기**
수만 줄의 JSON이나 로그를 대화창에서 파싱하게 하지 마세요. 파싱 스크립트를 짜달라고 한 뒤 로컬에서 실행하고, 결과 요약만 Claude에게 전달하세요.

**루트 CLAUDE.md에 모든 정보 때려 넣기**
CLAUDE.md가 길어질수록 Claude가 규칙을 무시하기 시작합니다. API 스펙 전체나 DB 스키마 같은 내용은 별도 파일로 빼고 폴더별 CLAUDE.md로 분리하세요.

**범위 없이 탐색 지시하기**
"이 버그 찾아줘"처럼 범위 없이 탐색을 시키면 수백 개의 파일을 읽으며 컨텍스트를 가득 채웁니다. 항상 탐색 범위를 명시하세요.

```
❌ "이 버그 어디서 나는지 찾아줘"
✅ "auth 모듈에서 로그인 시 발생하는 이 버그를 찾아줘"
```

**Claude가 같은 실수를 반복할 때 그냥 두기**
같은 실수가 반복된다면 CLAUDE.md가 너무 길거나 지시가 모호한 것입니다. 해당 규칙을 CLAUDE.md에 `IMPORTANT:` 키워드로 강조하거나, 더 구체적으로 다시 작성하세요.

**sudo로 npm 설치하기**
권한 오류 시 `sudo npm install -g`를 사용하지 마세요. 올바른 해결 방법은 `nvm`을 사용하는 것입니다.

## 💡 판단 기준 — 세션을 나눌 타이밍

아래 중 하나라도 해당되면 `/clear`하고 새 세션을 여세요.

- 현재 작업과 관련 없는 질문을 했다
- 기능 하나가 완전히 끝났다
- Claude가 이전에 없던 파일이나 패키지를 갑자기 만들기 시작한다
- 응답이 이상하게 느려지거나 엉뚱한 내용을 반복한다
