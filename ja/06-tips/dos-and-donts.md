# うまくいくこと & うまくいかないこと (Dos and Don'ts)

---

## ✅ 積極的に推奨すること (Dos)

**Plan Mode を先に**
`Shift + Tab`で常に計画を確認してからコードを修正させましょう。計画が間違っていると思ったら`ESC`で即座に中断する習慣もつけてください。

**TDD ベースの短いサイクル**
小さな変更 → lint/テスト → コミットの短いサイクルを維持してください。こうすることで、Claude が間違った方向に進んでも巻き戻しやすくなります。

**画像でコンテキストを伝える**
UIのトーン＆マナー、アーキテクチャ構成、デザインリファレンスはテキストより画面のスクリーンショットをドラッグ＆ドロップして伝えてください。テキストで説明するよりはるかに正確です。

**エラーログをそのまま渡す**
自分の解釈を外して、ターミナルのエラーログ全体をそのままコピー＆ペーストしてください。要約したり解釈を加えると、かえって情報が失われます。

**具体的な指示を出す**
曖昧な指示は曖昧な結果を生みます。「テストコードを書いて」よりも「ユーザーがログアウトしたときのエッジケースをカバーするテストコードを書いて。モックは使わないで。」のように具体的に指示してください。

**think hard を活用する**
複雑なアーキテクチャの決定やエッジケースが多い問題には、プロンプトの先頭に`think hard`を付けてください。Claude がより深く推論します。（内部の推論トークンが増えるため、本当に必要なときだけ使用してください）

```
"think hard about the edge cases in our authentication flow"
```

**作業前に探索を先に行う**
大きな作業を始める前に、Claude に関連ファイルをまず読ませ、何も修正しないよう明示してください。探索と実装を分けることで、엉뚱なファイルの修正を防げます。

```
"먼저 인증 관련 파일들을 읽고 구조를 파악해줘. 아직 아무것도 수정하지 마."
```

## ❌ 絶対に避けるべきこと (Don'ts)

**1つのセッションで複数機能を開発する**
コンテキストが汚染されてハルシネーションが発生します。1セッション＝1フィーチャーの原則を守り、機能が終わったら`/clear`してください。関係のない質問を途中に挟むことも同様です。

**大容量ファイルを直接解析させる**
数万行のJSONやログをチャット上でパースさせないでください。パーシングスクリプトを書いてもらってからローカルで実行し、結果の要約だけClaudeに渡してください。

**ルートの CLAUDE.md にすべての情報を詰め込む**
CLAUDE.md が長くなるほど、Claude がルールを無視し始めます。APIスペック全体やDBスキーマなどの内容は別ファイルに切り出し、フォルダごとのCLAUDE.mdに分散させてください。

**スコープなしで探索を指示する**
「このバグを見つけて」のようにスコープなしで探索させると、数百個のファイルを読んでコンテキストをいっぱいにしてしまいます。必ず探索範囲を明示してください。

```
❌ "이 버그 어디서 나는지 찾아줘"
✅ "auth 모듈에서 로그인 시 발생하는 이 버그를 찾아줘"
```

**Claude が同じミスを繰り返しているのに放置する**
同じミスが繰り返されるなら、CLAUDE.md が長すぎるか指示が曖昧すぎるかのどちらかです。該当ルールをCLAUDE.mdで`IMPORTANT:`キーワードで強調するか、より具体的に書き直してください。

**sudo で npm をインストールする**
権限エラーが出たときに`sudo npm install -g`を使わないでください。正しい解決方法は`nvm`を使うことです。→ [トラブルシューティング](../01-getting-started/04-troubleshooting.md)

## 💰 コストが急に増えたときのチェックリスト

コストの問題が発生したときは、順番に確認してください。

```
1. /context → System Tools の項目が大きければ MCP を無効化
2. /context → User/Assistant が大きければ /compact または /clear
3. CLAUDE.md の長さを確認 → 300行を超えたらフォルダごとに分割
4. サブエージェントが不要な作業なのに複数エージェントが動いていないか確認
5. 会話の途中に関係ない質問が多く挟まっていないか確認
```

→ コスト最適化戦略の全体ガイド: [cost-optimization.md](./cost-optimization.md)

## 💡 判断基準 — セッションを分けるタイミング

以下のどれか1つでも該当したら`/clear`して新しいセッションを開いてください。

- 現在の作業と関係ない質問をした
- 機能が1つ完全に終わった
- Claude がそれまでなかったファイルやパッケージを突然作り始めた
- 応答がおかしく遅くなったり、엉뚱な内容を繰り返したりする
- `/context`でコンテキストが70%以上埋まった
